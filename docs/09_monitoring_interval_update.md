# Monitoring Interval Update

## 1. 변경 목적

Ops Monitor의 백그라운드 모니터링 주기를 `60초`에서 `30초`로 조정했다.

이번 변경 목적은 다음과 같다.

- DB 장애 감지 시간을 줄이기
- 복구 알림 확인 대기 시간을 줄이기
- 대시보드에서 현재 모니터링 주기를 더 직관적으로 확인하기

## 2. 반영 파일

이번 변경은 아래 파일에 반영됐다.

| 파일 | 반영 내용 |
|---|---|
| `app/config.py` | 기본 모니터링 주기 fallback 값을 `30`으로 조정 |
| `.env` | 로컬 실행 환경의 `MONITOR_INTERVAL_SECONDS=30` 반영 |
| `.env.example` | 새 환경 생성 시 기본값을 `30초`로 맞춤 |
| `app/services/monitoring_loop.py` | 런타임 상태 표시 기본값을 `30초`로 정리 |

## 3. 기대 효과

### 3.1 장애 감지 속도 개선

기존 `60초` 주기에서는 DB 중지 후 실제 장애 이벤트가 대시보드와 알림에 반영되기까지 최대 1분 이상 걸릴 수 있었다.

`30초`로 줄이면:

- 장애 발생 감지 시간이 더 짧아짐
- 복구 후 `recovery` 이벤트 확인도 더 빠름
- 로컬 테스트 시 대기 시간이 줄어듦

### 3.2 운영 확인 흐름 단순화

장애 재현 테스트를 할 때 다음 흐름이 더 빠르게 확인된다.

1. DB 컨테이너 중지
2. 장애 이벤트 감지
3. DB 컨테이너 재시작
4. 복구 이벤트 감지

## 4. 주의할 점

모니터링 주기를 짧게 하면 감지 속도는 좋아지지만, 그만큼 상태 확인 호출 횟수는 늘어난다.

현재 프로젝트 기준에서는 다음 이유로 `30초`가 과하지 않다고 판단했다.

- 점검 대상이 로컬 DB와 시스템 자원 중심이라 호출 비용이 크지 않음
- 운영 규모가 큰 서비스가 아니라 학습/검증 목적 프로젝트임
- 대시보드와 알림 흐름을 눈으로 확인하기에 `30초`가 적당함

## 5. 반영 후 실행

### Docker Compose 실행 중일 때

```bash
docker-compose up -d --force-recreate app
```

`.env` 값 변경은 실행 중 컨테이너에 자동 반영되지 않기 때문에 앱 컨테이너 재생성이 필요하다.

### 로컬 uvicorn 실행일 때

`--reload`로 실행 중이면 저장 후 자동 반영될 수 있고, 아니라면 앱 재시작이 필요하다.

## 6. 확인 방법

대시보드 또는 보호된 상태 API에서 현재 주기를 확인할 수 있다.

- `/dashboard`
- `/monitoring/status`

기대값:

```json
{
  "interval_seconds": 30
}
```
