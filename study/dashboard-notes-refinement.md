# 대시보드와 메모 서비스 정제 스터디

## 왜 이 문서를 따로 남기는가

이번 작업은 단순히 화면을 예쁘게 바꾸는 일이 아니었습니다.

대시보드를 운영 화면처럼 보이게 만들고, `demo-notes`를 실제로 작동하는 부가 서비스처럼 붙이는 과정에서 "어디까지를 진짜 기능으로 볼 것인가"를 계속 판단해야 했습니다.

그래서 결과만 적는 문서와 별개로, 왜 그렇게 정리했는지와 실제로 어떤 장애를 밟았는지를 남기는 스터디 문서를 따로 둡니다.

## 1. 카드형 화면을 줄인 이유

처음 화면은 카드가 많은 전형적인 대시보드에 가까웠습니다.

이 구성은 빠르게 만들기 쉽지만 운영 화면처럼 보이기에는 한계가 있었습니다.

- 핵심 장애 상태와 참고용 수치가 비슷한 무게로 보임
- 움직이지 않는 숫자도 실제 지표처럼 보임
- 사용자가 어디부터 읽어야 하는지 판단하기 어려움

그래서 이번 정리는 "카드를 더 많이 놓는 것"이 아니라 "정보 위계를 다시 세우는 것"에 집중했습니다.

## 2. 왜 상태 다섯 개를 한 블록으로 묶었는가

운영자가 제일 먼저 보는 질문은 결국 하나입니다.

"지금 서비스가 정상인가?"

이 질문에 가장 직접적으로 연결되는 상태는 아래 다섯 가지였습니다.

- API 상태
- DB 상태
- 모니터링 상태
- 디스코드 알림 상태
- 설정 상태

이 다섯 가지를 한 줄 안에서 묶으면, 운영자가 화면을 스캔하듯 읽을 수 있습니다. 반대로 각각을 따로 떼어 놓으면 상태 판단보다 레이아웃 해석이 먼저 필요해집니다.

즉, 이 구성은 장식이 아니라 우선순위의 반영입니다.

## 3. 왜 메모리와 디스크는 도넛으로 보였는가

메모리와 디스크는 정상/오류보다 점유율을 읽는 정보에 가깝습니다.

그래서 상태형 카드에 섞기보다는 비율형 시각 요소로 분리하는 편이 더 자연스러웠습니다.

이때 기준은 화려함이 아니라 의미 적합성이었습니다.

- 상태형 정보는 텍스트 중심
- 비율형 정보는 도형 중심

이 기준을 지키면 화면이 덜 산만해지고, 왜 이 요소가 여기에 있는지도 설명하기 쉬워집니다.

## 4. 왜 더미 데이터는 빼야 했는가

운영 화면에서는 움직이지 않는 숫자나 근거 없는 카운트가 금방 티가 납니다.

처음에는 분위기를 만들 수 있어 보여도, 실제로는 신뢰를 깎습니다.

이번 작업에서 세운 기준은 단순했습니다.

- 실제로 바뀌는 값만 남긴다
- 아직 연결되지 않은 정보는 억지로 채우지 않는다
- 설명 문구보다 상태 판단을 우선한다

이 판단이 중요했던 이유는, 사용자에게 "연습용 화면"이 아니라 "운영 중인 화면"처럼 느껴지게 만드는 핵심이 바로 데이터의 진실성이었기 때문입니다.

## 5. 왜 `demo-notes`를 붙였는가

대시보드가 실제 운영 프로젝트처럼 보이려면, 감시 대상이 메인 API와 DB만 있어서는 조금 약했습니다.

그래서 사용자가 직접 들어가서 입력, 저장, 수정, 삭제를 해볼 수 있는 작은 부가 서비스를 붙였습니다.

메모 서비스를 선택한 이유는 명확했습니다.

- 브라우저에서 바로 조작할 수 있음
- 데이터 구조가 단순해 빠르게 붙일 수 있음
- DB 연동 여부를 설명하기 쉬움
- 대시보드가 실제로 감시하는 대상 서비스로 연결하기 좋음

즉, `demo-notes`는 장식이 아니라 운영 연결감을 강화하기 위한 장치였습니다.

## 6. 왜 DB 연동까지 갔는가

파일 저장이나 메모리 저장으로도 데모는 가능했습니다.

하지만 그렇게 두면 운영 프로젝트로서의 설득력이 약합니다.

DB에 붙이면서 다음 장점이 생겼습니다.

- 앱 상태와 저장소 상태를 분리해 설명할 수 있음
- DB 장애가 보조 서비스에도 영향을 주는 흐름을 보여줄 수 있음
- "저장된다"가 아니라 "어디에 저장되는가"를 말할 수 있음

특히 이 프로젝트에서는 메인 앱과 메모 서비스가 같은 PostgreSQL을 바라보므로, 운영 관점에서 장애 전파를 설명하기 쉬워졌습니다.

## 7. 실제로 밟은 장애와 배운 점

이번에 `demo-notes`가 안 열렸던 원인은 단순히 "데이터가 없어서"가 아니었습니다.

실제 원인은 두 층으로 나뉘어 있었습니다.

### 7.1 로컬 실행과 Compose 실행의 호스트 차이

컨테이너 내부에서는 DB host가 `db`여야 맞지만, 로컬에서 직접 실행할 때 `db`는 해석되지 않습니다.

이 차이를 무시하면 아래 같은 식의 오류를 만나게 됩니다.

- `No address associated with hostname`

그래서 현재 코드는 실행 위치를 보고 다음처럼 보정합니다.

- 컨테이너 내부: `db`
- 로컬 직접 실행: `localhost`

이 보정은 "편의 기능"이라기보다, 같은 프로젝트를 두 실행 방식에서 모두 다룰 수 있게 만드는 운영 편의 장치에 가깝습니다.

### 7.2 Compose 환경변수 치환 오류

더 직접적인 장애는 `notes` 컨테이너가 반복 재시작한 사건이었습니다.

원인은 `DEMO_NOTES_DATABASE_URL`에 들어가야 할 값이 실제 계정 정보로 치환되지 않고, `${POSTGRES_USER}` 같은 문자열 그대로 전달된 점이었습니다.

이 상태에서는 컨테이너가 뜨자마자 DB 인증에 실패하고 종료됩니다.

이 문제를 통해 확인한 기준은 분명합니다.

- `notes` 서비스에도 `env_file: .env`를 명시한다
- Compose에서 치환되어야 하는 값은 `${...}` 형태를 정확히 사용한다
- 컨테이너가 재시작 루프에 들어가면 브라우저 쪽에서는 `ERR_EMPTY_RESPONSE`처럼 보일 수 있다

즉, 브라우저 증상만 보고 프론트 문제로 단정하면 안 되고, 컨테이너 상태와 로그까지 같이 봐야 합니다.

## 8. 왜 빈 데이터와 연결 실패를 분리해야 했는가

또 하나 중요했던 포인트는 "메모가 없는 상태"와 "서비스가 죽은 상태"를 같은 화면으로 보이면 안 된다는 점이었습니다.

둘은 사용자 입장에서 완전히 다른 상황입니다.

- 메모가 없음: 정상 동작 중인 빈 상태
- 연결 실패: 운영자가 점검해야 하는 오류 상태

그래서 `demo-notes`는 지금 아래 기준으로 응답합니다.

- 데이터가 없으면 `저장된 메모가 없습니다.`
- API/DB 연결이 실패하면 오류 안내 표시
- 루트 페이지 `/`는 가능한 한 응답 유지

이 기준을 넣고 나서야 비로소 "빈 서비스"와 "고장난 서비스"가 구분되기 시작했습니다.

## 9. 이번 작업으로 세운 기준

앞으로 비슷한 작업에서도 유지하고 싶은 기준은 다음과 같습니다.

- 운영 화면은 설명보다 상태 판단이 먼저여야 한다
- 더미 데이터는 분위기보다 신뢰를 더 크게 해칠 수 있다
- 부가 서비스는 실제로 조작 가능해야 의미가 있다
- 로컬 실행과 컨테이너 실행 차이는 코드와 문서에서 모두 다뤄야 한다
- 브라우저 오류는 컨테이너 로그까지 확인하고 판단해야 한다
- 문서는 결과뿐 아니라 판단 기준도 남겨야 다음 수정이 쉬워진다
