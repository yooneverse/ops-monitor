# Security Base

## 1. 학습 목적

Ops Monitor 프로젝트에 적용한 기본 보안 설정을 개념 수준부터 정리한다.

이 문서는 다음 항목을 이해하는 데 목적이 있다.

| 구분 | 정리 내용 |
|---|---|
| 민감정보 관리 | `.env`, `.env.example`, `.gitignore` 역할 |
| API 접근 제어 | CORS 제한 이유 |
| Reverse Proxy 보안 | Nginx 헤더 적용 목적 |
| 오류 응답 관리 | 상세 예외를 숨기는 이유 |
| 문서화 기준 | 예시값과 플레이스홀더 사용 원칙 |

---

## 2. 왜 보안 기본 설정이 필요한가

프로젝트가 작아도 운영 구성 문서와 설정 파일에는 민감정보가 쉽게 섞일 수 있다.

예를 들어 다음 정보는 그대로 노출되면 위험할 수 있다.

| 항목 | 위험 |
|---|---|
| DB 계정명 | 인증 정보 노출 |
| DB 비밀번호 | 직접적인 보안 사고 가능 |
| 전체 `DATABASE_URL` | 계정 정보와 접근 경로 동시 노출 |
| 로컬 사용자 경로 | 개인 정보 노출 |
| 상세 DB 오류 원문 | 내부 구조 노출 |

따라서 코드 구현과 별개로 기본 보안 기준을 먼저 세워둘 필요가 있다.

---

## 3. `.env`와 `.env.example`의 차이

### 3.1 `.env`

`.env`는 실제 실행 환경에서 사용하는 민감한 설정 파일이다.

예시:

```env
POSTGRES_USER=<DB_USER>
POSTGRES_PASSWORD=<DB_PASSWORD>
POSTGRES_DB=<DB_NAME>
DATABASE_URL=postgresql://<DB_USER>:<DB_PASSWORD>@db:5432/<DB_NAME>
```

실제 값이 들어가기 때문에 Git에 올리면 안 된다.

---

### 3.2 `.env.example`

`.env.example`은 팀원이나 평가자가 어떤 환경변수가 필요한지 알 수 있도록 제공하는 예시 파일이다.

| 파일 | 역할 |
|---|---|
| `.env` | 실제 실행값 저장 |
| `.env.example` | 필요한 키 목록과 형식 안내 |

---

## 4. `.gitignore`가 중요한 이유

`.gitignore`는 Git이 추적하지 않을 파일을 정의한다.

Ops Monitor에서는 다음 기준이 중요하다.

```gitignore
.env
.env.*
!.env.example
```

의미는 다음과 같다.

| 설정 | 의미 |
|---|---|
| `.env` | 실제 환경파일 제외 |
| `.env.*` | 변형된 환경파일도 제외 |
| `!.env.example` | 예시 파일은 예외적으로 포함 |

이 규칙이 없으면 실수로 실제 비밀번호가 GitHub에 올라갈 수 있다.

---

## 5. CORS를 제한하는 이유

CORS는 브라우저 기반 요청에서 다른 출처의 접근을 통제하는 정책이다.

Ops Monitor는 로컬 대시보드와 조회 API 중심이므로 허용 범위를 최소화했다.

```python
allow_origins=["http://localhost", "http://127.0.0.1"]
allow_methods=["GET"]
```

| 항목 | 이유 |
|---|---|
| 로컬 출처만 허용 | 불필요한 외부 출처 접근 제한 |
| GET 중심 허용 | 현재 기능이 조회 위주이기 때문 |

보안 설정은 넓게 열기보다 필요한 범위만 허용하는 것이 기본 원칙이다.

---

## 6. 왜 상세 오류를 그대로 보여주지 않는가

개발 중에는 예외 메시지가 도움이 되지만, 운영 응답에서 내부 오류를 그대로 노출하면 시스템 구조를 유추할 수 있다.

Ops Monitor는 DB 연결 실패 시 다음 고정 메시지를 사용한다.

```text
Database connection failed
```

| 방식 | 특징 |
|---|---|
| 상세 예외 노출 | 디버깅에는 도움, 운영 노출 위험 큼 |
| 고정 메시지 사용 | 정보 노출 최소화, 응답 일관성 유지 |

---

## 7. 보안 헤더의 의미

Nginx에서 적용한 보안 헤더는 브라우저 동작을 좀 더 안전한 방향으로 제한한다.

| Header | 목적 |
|---|---|
| `X-Content-Type-Options` | 콘텐츠 타입 오해석 방지 |
| `X-Frame-Options` | 클릭재킹 방지 |
| `X-XSS-Protection` | 구형 브라우저 XSS 필터 사용 |
| `Referrer-Policy` | 외부 이동 시 참조 정보 제한 |

이 헤더들은 대규모 보안 체계의 전부는 아니지만, 기본 보안 기준으로 의미가 있다.

---

## 8. 문서에서의 민감정보 처리 기준

문서에는 실제 값을 적지 않고 플레이스홀더를 사용한다.

| 항목 | 문서 표기 방식 |
|---|---|
| DB 사용자명 | `<DB_USER>` |
| DB 비밀번호 | `<DB_PASSWORD>` |
| DB 이름 | `<DB_NAME>` |
| DB 호스트 | `<DB_HOST>` 또는 `db` |
| DB 포트 | `<DB_PORT>` 또는 공개 가능한 기본 포트 |
| 연결 문자열 | `postgresql://<DB_USER>:<DB_PASSWORD>@<DB_HOST>:<DB_PORT>/<DB_NAME>` |

문서가 길어질수록 이런 기준이 더 중요해진다.

---

## 9. 프로젝트 적용 정리

| 항목 | 적용 내용 |
|---|---|
| 환경변수 분리 | `.env`, `.env.example` 분리 |
| Git 제외 | `.gitignore`에 환경파일 제외 규칙 적용 |
| CORS 제한 | 로컬 출처 + GET 중심 제한 |
| 보안 헤더 | Nginx에서 공통 적용 |
| 오류 메시지 | DB 상세 예외 비노출 |
| 문서 기준 | 플레이스홀더 기반 표기 |

---

## 10. 정리

어제 작업을 통해 보안은 별도 인증 기능이 있어야만 시작되는 것이 아니라, 환경변수 분리, CORS 제한, 헤더 설정, 오류 메시지 관리처럼 기본적인 설정 단계부터 시작된다는 점을 확인했다.

작은 프로젝트라도 이런 기준을 문서와 설정 파일에 함께 남겨두면, 이후 기능이 늘어나도 보안 기준을 일관되게 유지하기 쉬워진다.
