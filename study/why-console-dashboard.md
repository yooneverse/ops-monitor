# 왜 대시보드를 콘솔형으로 정리했는가

## 개요

이 문서는 Ops Monitor 대시보드를 일반적인 마케팅식 카드 화면이 아니라, 콘솔형 운영 화면으로 정리한 이유를 설명하기 위해 작성했습니다.

여기서 말하는 콘솔형은 단순히 어두운 테마나 개발자 느낌의 디자인을 뜻하지 않습니다.

이번 프로젝트에서 콘솔형은 다음 기준에 더 가깝습니다.

- 상태를 빠르게 훑을 수 있을 것
- 정보의 우선순위가 화면 구조에 반영될 것
- 보이는 요소가 실제 운영 판단에 도움이 될 것
- 보기 좋은 데모보다 점검 가능한 화면에 가까울 것

즉, "관리자처럼 보이는 디자인"보다 "관리자가 실제로 읽기 쉬운 구조"를 우선한 선택이었습니다.

## 1. 왜 일반 카드형 대시보드가 맞지 않았는가

처음 대시보드를 다듬기 전에는 많은 운영 화면이 그렇듯 카드들을 나란히 두는 방식이 자연스러워 보였습니다.

하지만 이 프로젝트에서는 그 방식이 몇 가지 이유로 잘 맞지 않았습니다.

- API 상태, DB 상태, 설정 상태처럼 우선순위가 높은 항목과 보조 지표가 같은 크기로 보임
- 카드가 많아질수록 사용자가 무엇을 먼저 봐야 하는지 스스로 해석해야 함
- 더미 숫자나 움직이지 않는 정보도 그럴듯한 지표처럼 보이기 쉬움
- 화면은 풍성해 보이지만 실제 판단 속도는 느려짐

운영 화면에서 중요한 것은 "요소 개수"보다 "판단 순서"입니다.

그래서 이번 대시보드는 무엇을 더 넣을지보다, 어떤 순서로 읽히게 할지를 먼저 정리했습니다.

## 2. 왜 콘솔형이 이 프로젝트와 잘 맞았는가

Ops Monitor는 기본적으로 아래 질문에 빠르게 답해야 하는 프로젝트입니다.

1. 서비스가 살아 있는가
2. 어디가 문제인가
3. 지금 당장 확인하거나 눌러볼 곳이 어디인가

이 세 질문은 마케팅용 요약 화면보다 운영 콘솔 구조에 더 잘 맞습니다.

콘솔형으로 정리하면 자연스럽게 다음 흐름을 만들 수 있습니다.

- 상단에서 핵심 상태를 한 번에 확인
- 좌측 메뉴나 검색으로 필요한 영역을 바로 찾기
- 최근 알림과 경고 흐름으로 문제 맥락 확인
- 보조 서비스나 운영 액션으로 이어지기

즉, 콘솔형은 취향 문제가 아니라 프로젝트 목적과 읽기 방식이 더 잘 맞는 형태였습니다.

## 3. 왜 상단 상태를 한 묶음으로 봐야 했는가

운영자가 가장 먼저 보는 것은 화면의 개성이 아니라 현재 상태입니다.

그래서 API, DB, 모니터링, 디스코드 알림, 설정 상태를 분산시키지 않고 상단의 하나의 상태 구역으로 묶었습니다.

이렇게 한 이유는 분명합니다.

- 핵심 상태를 위에서 아래로 찾지 않아도 됨
- 장애 징후를 한 번에 스캔할 수 있음
- 상태와 참고 지표를 분리하기 쉬움

콘솔형 구조에서는 이처럼 "가장 먼저 봐야 하는 것"이 가장 먼저 보이는지가 중요합니다.

예쁘게 흩어진 화면보다, 빠르게 읽히는 화면이 더 운영적이라고 판단했습니다.

## 4. 왜 보조 지표는 분리했는가

메모리와 디스크는 중요하지만, API나 DB처럼 바로 장애 여부를 판단하는 상태와는 성격이 다릅니다.

이 지표들은 비율을 읽는 정보에 가깝기 때문에, 상태 카드와 같은 층위에 섞어두면 오히려 핵심 상태를 흐립니다.

그래서 콘솔형 구조에서는 다음처럼 구분했습니다.

- 핵심 상태: 텍스트 중심 상태 블록
- 자원 지표: 도넛 차트 중심 보조 구역

이 분리는 디자인 다양성을 위한 것이 아니라, 서로 다른 정보를 같은 문법으로 보여주지 않기 위한 결정이었습니다.

## 5. 왜 사이드바와 검색이 실제로 동작해야 했는가

콘솔형 화면은 레이아웃만 비슷하다고 완성되지 않습니다.

왼쪽 사이드바와 검색창이 있어도 실제로 아무 일도 일어나지 않으면, 오히려 연습용 화면처럼 보입니다.

그래서 이번 기준에서는 최소한 다음 중 하나는 충족해야 했습니다.

- 메뉴를 누르면 다른 성격의 화면이나 구역으로 이동할 것
- 같은 화면 안에서라도 메뉴와 검색이 실제 필터 역할을 할 것

이번 단계에서는 구조를 과하게 키우지 않기 위해, 같은 페이지 안에서 검색과 구역 필터가 동작하도록 정리했습니다.

이런 상호작용이 들어가야 비로소 콘솔형 화면의 설득력이 생깁니다.

## 6. 왜 설명 문구를 줄였는가

운영 화면에서 긴 설명 문장은 친절해 보일 수는 있어도, 상태 판단 속도를 늦출 때가 많습니다.

특히 이번 프로젝트처럼 작은 운영 대시보드에서는 아래 원칙이 더 적합했습니다.

- 먼저 상태를 보여준다
- 필요한 경우에만 짧게 이유를 붙인다
- 장황한 소개 문구는 줄인다

그래서 운영 브리핑처럼 화면을 "설명하는 문장"은 줄이고, 상태와 로그 흐름을 직접 읽는 쪽으로 정리했습니다.

콘솔형은 말이 많은 화면이 아니라, 필요한 정보를 앞세우는 화면에 더 가깝다고 봤습니다.

## 7. 왜 더미 데이터와 정적인 숫자를 경계했는가

콘솔형 화면을 지향할수록 더미 데이터는 더 위험해집니다.

단순 카드 화면에서는 분위기를 위한 숫자가 어느 정도 넘어갈 수 있어도, 운영 콘솔처럼 보이기 시작하면 사용자는 숫자가 실제로 바뀌는지부터 보게 됩니다.

그때 정적인 값이 남아 있으면 바로 신뢰가 무너집니다.

그래서 이번 작업에서는 다음 기준을 세웠습니다.

- 실제로 갱신되지 않는 카운트는 제거
- 근거 없는 연결 수나 임의 값은 배제
- 데이터가 비어 있으면 비어 있다고 보여주기
- 연결 실패는 실패라고 보여주기

이 기준은 화면을 덜 화려하게 만들지만, 콘솔형 운영 화면으로서는 오히려 더 자연스럽습니다.

## 8. 왜 `demo-notes` 같은 부가 서비스가 콘솔형 구조와 잘 맞는가

콘솔형 대시보드는 감시 대상이 분명할수록 설득력이 높아집니다.

`demo-notes`를 붙인 이유도 여기에 있습니다.

대시보드가 메인 API와 DB만 보는 화면이면 다소 연습용처럼 느껴질 수 있습니다. 반면 실제로 접속해서 입력, 수정, 삭제할 수 있는 작은 서비스를 붙이면, 대시보드가 감시하는 대상이 더 구체적으로 보입니다.

즉, 콘솔형 구조는 단순히 "예쁜 관리자 화면"을 만드는 방식이 아니라, 운영 대상의 관계를 화면에 드러내는 방식이기도 합니다.

## 9. 이번 프로젝트에서 콘솔형이 의미했던 것

이번 프로젝트에서 콘솔형이라는 말은 대략 다음 의미를 가졌습니다.

- 한 번에 상황 파악이 가능할 것
- 상태와 지표의 층위가 섞이지 않을 것
- 더미보다 실제 연결성을 우선할 것
- 메뉴와 버튼이 실제 동작할 것
- 실패 상태도 화면 안에서 드러날 것

결국 콘솔형으로 정리한 이유는 "멋있어 보여서"가 아니라, Ops Monitor가 운영 프로젝트처럼 읽히도록 만들기 위해서였습니다.

## 10. 앞으로 유지할 기준

앞으로 대시보드를 더 손보더라도 아래 기준은 유지하는 편이 좋다고 봅니다.

- 핵심 상태는 항상 가장 먼저 읽혀야 한다
- 설명보다 판단이 먼저여야 한다
- 보조 지표는 핵심 상태와 섞지 않는다
- 더미 데이터로 분위기를 만들지 않는다
- 콘솔형이라는 이름보다 실제 운영 읽기 경험을 우선한다

이 기준이 유지되면 화면이 조금 달라져도 여전히 "운영 콘솔다운 대시보드"로 설명할 수 있습니다.
