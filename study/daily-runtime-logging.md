# Daily Runtime Logging

## 1. 목표

이 문서는 Ops Monitor 프로젝트에 추가한 일별 로그 저장 구조를 학습하기 위한 문서다.

이번 변경의 핵심 목표는 아래와 같다.

- 앱이 실행되는 동안 날짜별 로그 파일이 자동으로 남게 만들기
- 장애 이벤트를 메모리뿐 아니라 파일로도 남기기
- 사람이 읽기 쉬운 일일 리포트까지 함께 만들기
- Docker 환경에서도 호스트에서 로그를 바로 확인할 수 있게 하기

---

## 2. 왜 일별 로그가 필요한가

콘솔에만 로그가 출력되면 아래 문제가 생긴다.

- 컨테이너 재생성 시 과거 로그 추적이 불편하다
- 날짜 기준으로 장애 이력을 구분하기 어렵다
- 운영 점검 내용을 나중에 학습 자료로 되돌아보기 어렵다

특히 운영/인프라 관점에서는 "로그가 출력된다"보다 "언제, 어디에, 어떤 단위로 남는가"가 더 중요하다.

---

## 3. 이번에 추가한 로그 구조

이번 변경에서는 `logs/` 아래에 세 종류의 파일이 자동 생성된다.

### 3.1 애플리케이션 로그

경로 예시:

```text
logs/application/2026-07-10.log
```

포함 내용:

- 헬스체크 결과 로그
- 모니터링 루프 시작 로그
- 주기적 모니터링 사이클 완료 로그
- 예외 발생 로그

### 3.2 이벤트 원본 로그

경로 예시:

```text
logs/events/2026-07-10.jsonl
```

포함 내용:

- `incident`
- `recovery`
- `resource_alert`
- `resource_recovery`
- `notification_error`
- `monitoring_error`

`jsonl`은 JSON Lines 형식이라 한 줄에 이벤트 한 건씩 저장된다.  
이 형식은 나중에 스크립트나 분석 도구로 읽기 좋다.

### 3.3 사람이 읽기 쉬운 일일 이벤트 리포트

경로 예시:

```text
logs/reports/2026-07-10.md
```

이 파일은 Markdown 표 형식으로 같은 날짜의 이벤트를 누적한다.

즉,

- 머신 친화적인 원본: `jsonl`
- 사람 친화적인 요약: `md`

구조를 동시에 가지게 된다.

---

## 4. 어떤 흐름으로 저장되는가

### 4.1 일반 애플리케이션 로그

`uvicorn.error`, `uvicorn.access` 로거에 날짜별 파일 핸들러를 붙였다.

이 핸들러는 로그가 기록될 때마다 현재 날짜를 보고 아래 위치에 append 한다.

```text
logs/<stream-name>/YYYY-MM-DD.log
```

예:

```text
logs/application/2026-07-10.log
logs/access/2026-07-10.log
```

### 4.2 알림 이벤트 로그

기존에는 `alert_history`에 최근 이력만 메모리에 저장했다.  
이번에는 `add_alert_history()`가 호출될 때 파일에도 함께 기록되도록 바꿨다.

이렇게 하면 애플리케이션 재시작 후에도 날짜별 이벤트 파일은 남는다.

---

## 5. 모니터링 루프 로그를 추가한 이유

단순히 장애가 있을 때만 로그를 남기면 조용한 날에는 파일이 비어 있을 수 있다.  
그러면 "매일 로그가 작성된다"는 요구를 충족했다고 보기 어렵다.

그래서 모니터링 루프가 한 번 돌 때마다 아래 로그를 남기도록 했다.

```text
Monitoring cycle completed: db=connected memory=43.2% disk=51.8%
```

이 로그의 장점:

- 그날 앱이 실제로 돌아갔는지 확인 가능
- 장애가 없더라도 운영 흔적이 남음
- 메모리/디스크 수준을 일자별로 대략 추적 가능

---

## 6. Docker Compose에서 왜 `./logs:/app/logs`를 마운트했는가

컨테이너 내부 파일만 쓰면 컨테이너 재생성 시 로그를 잃기 쉽다.  
그래서 앱 서비스에 아래 볼륨을 연결했다.

```yaml
volumes:
  - ./logs:/app/logs
```

이렇게 하면:

- 컨테이너 안에서는 `/app/logs`
- 호스트에서는 `./logs`

같은 데이터를 보게 된다.

즉 운영 확인과 학습 복기가 쉬워진다.

---

## 7. 테스트에서 무엇을 검증했는가

이번에는 `unittest`로 아래 내용을 검증했다.

- 날짜와 stream 이름에 맞는 파일 경로가 만들어지는가
- 이벤트 저장 시 `jsonl`과 `md`가 함께 생성되는가
- 날짜별 로그 핸들러가 실제 파일에 로그를 쓰는가

이 검증은 "파일이 생긴다"는 사실뿐 아니라 "날짜 기준으로 분리된다"는 핵심 요구를 보호한다.

---

## 8. 운영 포트폴리오 관점에서 좋은 이유

이 구조는 단순한 콘솔 출력보다 훨씬 운영 친화적이다.

좋은 점:

- 날짜별 추적이 가능하다
- 장애 이벤트를 재시작 이후에도 보존한다
- 사람용 리포트와 머신용 원본을 동시에 남긴다
- Docker 환경에서도 로그 경로가 명확하다

면접에서 설명하기 좋은 문장:

- "모니터링 이벤트를 메모리만이 아니라 일자별 JSONL과 Markdown 리포트로 남기도록 설계했습니다."
- "장애가 없더라도 운영 흔적이 남도록 모니터링 사이클 로그를 기록했습니다."
- "컨테이너 내부 로그가 사라지지 않도록 호스트 볼륨에 매핑했습니다."
