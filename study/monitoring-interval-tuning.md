# Monitoring Interval Tuning

## 1. 왜 주기를 조정했는가

모니터링 프로젝트에서 주기는 단순한 숫자 설정이 아니라, 감지 속도와 시스템 부담 사이의 균형을 잡는 기준이다.

이번 프로젝트에서는 `60초`에서 `30초`로 조정했다.

이유는 다음과 같다.

- 장애 재현과 복구 확인이 너무 오래 걸리지 않게 하기 위해
- 대시보드와 알림 흐름을 학습하기 좋은 속도로 맞추기 위해
- DB 중지/복구 테스트를 더 짧은 루프로 반복하기 위해

## 2. 주기를 줄이면 좋아지는 점

### 빠른 장애 감지

DB를 중지했을 때 다음 모니터링 사이클에서 더 빨리 `incident`를 만들 수 있다.

### 빠른 복구 확인

DB를 다시 켠 뒤 `recovery` 이벤트가 오는 대기 시간도 줄어든다.

### 테스트 반복 속도 향상

학습 프로젝트에서는 "장애 발생 -> 복구 -> 알림 확인" 흐름을 여러 번 반복해보는 것이 중요하다.  
주기가 너무 길면 확인 시간이 길어져 실습 효율이 떨어진다.

## 3. 주기를 줄일 때 같이 생각할 점

### 호출 횟수 증가

주기를 절반으로 줄이면 상태 확인 호출 수는 두 배가 된다.

### 노이즈 증가 가능성

더 자주 돌수록 로그도 더 자주 남는다.

### 상태 전이 테스트 방식

복구 알림은 보통 아래처럼 두 번의 감지가 필요하다.

1. 장애 상태 감지
2. 정상 상태 감지

그래서 복구 테스트에서는 "내렸다가 바로 올리는 것"보다 각 상태가 한 번씩 실제 사이클에 잡히도록 기다리는 것이 중요하다.

## 4. 이 프로젝트에서 30초가 적당한 이유

현재 Ops Monitor는 다음 특성을 가진다.

- 단일 DB 상태 확인 중심
- 로컬 또는 소규모 Compose 기반 실행
- 학습 목적의 대시보드/알림 검증 비중이 큼

이런 조건에서는 `30초`가 너무 짧지도, 너무 길지도 않은 값이다.

- `10초`: 너무 잦아서 테스트 노이즈가 늘 수 있음
- `60초`: 장애/복구 확인 대기가 길어짐
- `30초`: 체감 가능한 응답성과 과도하지 않은 호출 수의 중간값

## 5. 실무적으로 배운 점

모니터링 주기는 "기본값 그대로 두는 설정"이 아니라, 프로젝트 목적에 맞춰 근거 있게 조정해야 하는 값이다.

이번 변경으로 남길 수 있는 포인트는 다음과 같다.

- 운영성 관점에서 감지 시간과 비용을 함께 고려했다
- 로컬 테스트 경험을 개선하기 위해 설정을 조정했다
- 코드, 예시 환경 파일, 런타임 표시값을 함께 맞췄다

이런 정리는 프로젝트를 단순 구현물이 아니라, 설정 의도까지 설명할 수 있는 작업물로 보이게 해준다.
