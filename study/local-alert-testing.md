# Local Alert Testing

## 1. 학습 목적

Ops Monitor 프로젝트에서 로컬 환경 기준으로 재현 가능한 장애 테스트 범위를 정리한다.

이 문서는 다음 내용을 빠르게 이해하는 것을 목표로 한다.

| 구분 | 정리 내용 |
|---|---|
| 테스트 대상 | DB 연결, Webhook 전송, 자원 임계치, 복구 알림 |
| 로컬 가능 범위 | 애플리케이션 레벨과 컨테이너 레벨 장애 |
| 확인 API | `/health`, `/alerts`, `/monitoring/status`, `/dashboard` |
| 운영 포인트 | 중복 알림 방지, 복구 알림, 전송 실패 처리 |

---

## 2. 왜 로컬 장애 테스트가 필요한가

장애 대응 프로젝트는 기능 구현만으로는 설득력이 부족하다.  
실제로 어떤 장애를 만들 수 있고, 감지 결과가 어디에 어떻게 반영되는지 보여주는 것이 중요하다.

로컬에서도 다음 범위는 충분히 검증할 수 있다.

| 항목 | 설명 |
|---|---|
| DB 연결 실패 | 잘못된 연결 정보 또는 컨테이너 중지 |
| 알림 전송 실패 | 잘못된 Discord Webhook URL |
| 자원 경고 | 낮은 임계치 설정으로 경고 유도 |
| 복구 알림 | 장애 상태 해소 후 recovery 확인 |
| 중복 알림 방지 | 같은 장애 상태를 유지하며 반복 전송 여부 확인 |

---

## 3. 로컬에서 가능한 장애 시나리오

### 3.1 DB 연결 실패

가장 기본적인 테스트다. `.env`의 `DATABASE_URL` host, password, db name 중 하나를 의도적으로 틀리게 바꾸면 된다.

```env
DATABASE_URL=postgresql://<DB_USER>:wrong_password@db:5432/<DB_NAME>
```

기대 결과는 다음과 같다.

| 확인 항목 | 기대 결과 |
|---|---|
| `/health` | `database.status = disconnected` |
| `/alerts` | `incident` 추가 |
| `/monitoring/status` | background task는 유지 |
| Discord | 장애 알림 전송 |

복구 후에는 `recovery` 이벤트가 추가되어야 한다.

---

### 3.2 Discord Webhook 전송 실패

장애 감지는 정상인데 알림만 실패하는 시나리오도 운영 관점에서 중요하다.

```env
DISCORD_WEBHOOK_URL=https://discord.com/api/webhooks/invalid
```

이 상태에서 DB 장애나 임계치 초과를 다시 만들면 된다.

| 확인 항목 | 기대 결과 |
|---|---|
| 장애 감지 | 정상 동작 |
| Discord 전송 | 실패 |
| `/alerts` | `notification_error` 기록 |
| 앱 상태 | 종료되지 않고 계속 동작 |

---

### 3.3 메모리 및 디스크 임계치 초과

실제 서버 자원을 고갈시키지 않아도, threshold를 낮추면 경고 흐름을 확인할 수 있다.

```env
MEMORY_ALERT_THRESHOLD=1
DISK_ALERT_THRESHOLD=1
```

| 확인 항목 | 기대 결과 |
|---|---|
| `/alerts` | `resource_alert` 기록 |
| Discord | warning 알림 전송 |
| Dashboard | 최근 알림 영역에 표시 |

임계치를 다시 정상값으로 돌리면 `resource_recovery`가 발생해야 한다.

---

### 3.4 중복 알림 방지

같은 장애를 계속 유지했을 때 루프마다 같은 알림을 반복 전송하면 운영 품질이 떨어진다.

테스트 방법은 다음과 같다.

1. DB 연결 실패 또는 임계치 초과 상태를 만든다.
2. 모니터링 주기를 2~3회 이상 지나도록 유지한다.
3. 알림 이력과 Discord 전송 횟수를 확인한다.

기대 결과는 다음과 같다.

| 항목 | 기대 결과 |
|---|---|
| 최초 장애 | 1회 전송 |
| 동일 상태 유지 | 추가 반복 전송 없음 |
| 복구 시점 | recovery 1회 전송 |

---

## 4. 확인 API

로컬 테스트 시 주로 확인하는 API는 다음과 같다.

| Endpoint | 목적 |
|---|---|
| `/health` | API와 DB 상태 확인 |
| `/alerts` | 최근 알림 이력 확인 |
| `/monitoring/status` | 모니터링 루프 동작 상태 확인 |
| `/dashboard` | UI 기준 상태 반영 확인 |

---

## 5. 로컬 환경의 한계

로컬에서도 운영 로직 검증은 가능하지만, 실제 인프라 장애 전체를 재현할 수는 없다.

| 로컬에서 어려운 항목 | 이유 |
|---|---|
| 멀티 노드 네트워크 장애 | 단일 환경 위주 |
| 로드밸런서 장애 | 별도 인프라 미구성 |
| 보안 그룹/방화벽 이슈 | 클라우드 레벨 설정 부재 |
| 실제 디스크 고갈 | 시스템 전체 영향 위험 |

따라서 로컬 테스트는 “운영 로직 검증”, 실제 인프라 테스트는 “배포 환경 검증”으로 구분하는 것이 적절하다.

---

## 6. 프로젝트 적용 포인트

| 항목 | 적용 내용 |
|---|---|
| DB 초기 장애 감지 | 첫 체크에서도 `incident` 발생 |
| 자원 복구 알림 | `resource_recovery` 추가 |
| Webhook 실패 이력 | `notification_error` 기록 |
| 모니터링 상태 갱신 | `.env` 기반 설정 재확인 |
| UI 반영 | Dashboard에 신규 알림 타입 스타일 추가 |

---

## 7. 정리

이번 정리를 통해 로컬 환경에서도 단순 컨테이너 중지/재시작 외에 DB 연결 오류, 전송 실패, 임계치 초과, 복구, 중복 알림 방지까지 충분히 검증할 수 있음을 확인했다.

즉, 로컬 환경은 한계가 있지만 운영형 알림 로직을 설명하고 증명하는 데에는 충분한 실습 범위를 제공한다.
